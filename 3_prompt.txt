(1) IDEMPOTENCY KEYS (PRIORIDAD MÁXIMA)
----------------------------------------
Problema: retries y reintentos pueden ejecutar 2 veces una acción WRITE (transfer, create ticket, etc).

Requerimientos:
- Aceptar `Idempotency-Key` como header en `POST /v1/run`.
- En MCP `tools/call`, aceptar un campo meta (ej `idempotency_key`) o header si MCP va sobre HTTP; definilo y documentalo.
- Definir formalmente el “efecto” de la tool:
  - Si ya hay read/write, usalo.
  - Si no, agregar campo `effect` en `tools`: `"read"|"write"`.
  - Regla: GET => read, otros => write (fallback).
- Persistencia:
  - Nueva tabla `idempotency_keys`:
    - `id` UUID PK
    - `org_id` UUID NOT NULL
    - `tool_name` TEXT NOT NULL
    - `idempotency_key` TEXT NOT NULL
    - `request_fingerprint` TEXT NOT NULL
    - `status` TEXT NOT NULL (IN_PROGRESS|COMPLETED|FAILED)
    - `response_redacted_json` JSONB NULL
    - `created_at` TIMESTAMPTZ NOT NULL default now()
    - `expires_at` TIMESTAMPTZ NOT NULL
  - Uniq: `(org_id, tool_name, idempotency_key)`
  - Index: `(expires_at)`, `(org_id, tool_name, created_at desc)`
- request_fingerprint:
  - Hash determinístico (SHA-256) de:
    - tool_name
    - canonical_json(input)
    - actor (y opcional role/scopes si se considera parte del “quién”)
  - Debe ser estable (mismo input -> mismo fingerprint).
- Comportamiento exacto:
  - Si la tool es WRITE:
    - Si falta Idempotency-Key:
      - NO romper compatibilidad: permitir.
      - Pero si policy tiene `require_idempotency=true` entonces denegar con `IDEMPOTENCY_REQUIRED`.
    - Si viene Idempotency-Key:
      1) Buscar registro por (org_id, tool_name, idempotency_key)
      2) Si existe:
         - Si fingerprint coincide y status=COMPLETED:
           - Devolver response cacheada
           - Marcar outcome = REPLAY
         - Si fingerprint coincide y status=IN_PROGRESS:
           - Devolver `IDEMPOTENCY_IN_PROGRESS` (HTTP 409) o 202 con estado; definí uno y documentalo.
         - Si fingerprint NO coincide:
           - Error `IDEMPOTENCY_KEY_CONFLICT` (409)
      3) Si NO existe:
         - Insertar status=IN_PROGRESS
         - Ejecutar flujo normal (schema/policy/dlp/egress/execute/audit)
         - Al finalizar: actualizar a COMPLETED y guardar response_redacted_json (o FAILED + error code)
- Auditoría:
  - Agregar campos:
    - `idempotency.present` boolean
    - `idempotency.outcome` enum: NEW|REPLAY|IN_PROGRESS|CONFLICT|MISSING_REQUIRED|SKIPPED_NOT_WRITE
  - No registrar idempotency_key en claro en logs. En auditoría, como mucho `present` y `outcome`.
- Cleanup:
  - TTL configurable (ej 24h por env var).
  - Agregar comando: `cmd/cleanup-idempotency` que borra expirados.
  - NO crear cron interno automático si no existe ya; solo comando y doc.

Tests:
- Unit: canonical JSON y fingerprint stable; conflict; replay.
- Integration/E2E:
  - Ejecutar 2 veces misma call con Idempotency-Key => 1 sola ejecución real del mock-tool (podés medir por contador en mock-tools o audit).
  - Conflict: mismo key distinto input => 409 + code IDEMPOTENCY_KEY_CONFLICT.
  - In-progress: simular ejecución lenta + segundo request => devuelve in-progress.

Errores nuevos:
- `IDEMPOTENCY_REQUIRED` (400 o 409, definilo) cuando policy exige y falta.
- `IDEMPOTENCY_KEY_CONFLICT` (409).
- `IDEMPOTENCY_IN_PROGRESS` (409 o 202).
- `IDEMPOTENCY_STORE_ERROR` (500) si falla DB.

----------------------------------------
(2) TIMEOUT BUDGET END-TO-END
----------------------------------------
Problema: timeout por tool no alcanza; enterprise necesita budget total por request.

Requerimientos:
- Aceptar `X-Timeout-Ms` (header) en `/v1/run` (y MCP).
- Clamp:
  - min 1000ms
  - max configurable (ej 30000ms)
  - default configurable (ej 10000ms)
- Implementar “Budget” (en un paquete interno) con:
  - `Start()`, `Remaining()`, `Consume(stage, duration)` best-effort
  - Guardar `stage_durations_ms` (map) para auditoría
- Enforcement:
  - Si Remaining() <= 0 antes de ejecutar => `TIMEOUT_BUDGET_EXCEEDED` (408)
  - El executor HTTP debe usar remaining budget como timeout (con mínimo 500ms).
  - Si se agota durante execute => `TIMEOUT` o `TIMEOUT_BUDGET_EXCEEDED` (definí bien: 
    - si timeout del http client dispara => TIMEOUT
    - si remaining se vuelve <=0 => TIMEOUT_BUDGET_EXCEEDED)
- Auditoría:
  - Guardar:
    - `timeout_ms`
    - `budget_remaining_ms_at_execute`
    - `stage_durations_ms` (jsonb)
- Logs/OTel:
  - Attribute `timeout_ms` y `budget_remaining_ms_at_execute`.

Tests:
- Unit: parse/clamp.
- E2E: mock-tool que duerme N ms; set X-Timeout-Ms < N; debe cortar con timeout.

Errores nuevos:
- `TIMEOUT_BUDGET_EXCEEDED` (408).

(3) AUDIT EXPORT SIEM-FRIENDLY
----------------------------------------
Problema: el buyer enterprise quiere export SIEM-friendly ya, sin WORM.

Requerimientos:
- Nuevo endpoint:
  - `GET /v1/audit/export`
  - Query:
    - `format=jsonl|csv` (default jsonl)
    - `from`, `to` (ISO8601)
    - `tool_name`, `decision`, `status`
    - `limit` (capado, default)
- Streaming:
  - NO cargar todo en memoria. Usar cursor/paginación.
- Contenido:
  - Incluye campos:
    - request_id, timestamp, org_id, actor, role, scopes (si no sensibles), tool_name
    - decision, status, latency_ms, error_code
    - redacted input/context/output
    - dlp_summary
    - prev_event_hash, event_hash, hash_algo
- Orden:
  - Por created_at asc (o id asc) consistente.
- CSV:
  - Campos planos. Para payloads redacted: o excluirlos o incluirlos como JSON string. Documentar decisión.
- Auth:
  - Igual que /v1/audit.

Tests:
- Integration: export jsonl válido; respeta filtros; incluye hash-chain.
- E2E: correr un run + export y verificar que aparece.

Errores nuevos:
- `EXPORT_FORMAT_INVALID` (400).
- `EXPORT_RANGE_INVALID` (400).
